# Фианит Снаб: заявки на довоз ТМЦ

Приложение для управления активной потребностью подразделений Фианит Ломбард к доставке расходных материалов водителем.

Главная рабочая сущность — `demand_lines`: сводный активный остаток по подразделению и позиции. Исторические заявки товароведов сохраняются отдельно и только увеличивают эту активную потребность.

## Стек

- Python 3.11+
- FastAPI
- SQLite
- SQLAlchemy ORM
- Jinja2
- Bootstrap 5
- uvicorn

## Установка

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## Запуск

```bash
uvicorn app.main:app --reload
```

Откройте в браузере:

```text
http://127.0.0.1:8000
```

SQLite-БД `sklad_requests.db` создаётся автоматически в корне проекта. Поданные заявки, активные потребности и визиты сохраняются в этом файле и остаются после перезапуска приложения. При старте приложение добавляет реальные подразделения из seed-списка, тестовых пользователей и справочник позиций, не удаляя уже поданные заявки.

Чтобы полностью сбросить данные, остановите сервер и удалите файл `sklad_requests.db`. При следующем запуске seed-данные будут созданы заново.

Для рабочих запусков задайте секрет сессий:

```bash
export SESSION_SECRET_KEY="сложная-случайная-строка"
```

Если `APP_ENV=production`, приложение не стартует без `SESSION_SECRET_KEY`.
Для HTTPS-запуска также можно включить secure-cookie:

```bash
export APP_ENV=production
export SESSION_COOKIE_SECURE=1
```

В production демо-пользователи с известными паролями не создаются. Если в базе ещё нет активного администратора, задайте начальный пароль:

```bash
export INITIAL_ADMIN_LOGIN="admin"
export INITIAL_ADMIN_PASSWORD="сложный-начальный-пароль"
```

В локальном dev-режиме без `APP_ENV=production` по-прежнему используется локальный dev-секрет и создаются демо-пользователи из таблицы ниже. Чтобы явно включить демо-пользователей в другом окружении, задайте `SEED_DEMO_USERS=true`.

## Перенос и резервная копия

Все рабочие данные сохраняются в файле `sklad_requests.db` в корне проекта:

- подразделения;
- пользователи;
- логины;
- bcrypt-хеши паролей;
- заявки;
- активная потребность;
- визиты водителей;
- результаты доставок;
- справочник позиций и цены.

Чтобы перенести систему на другой компьютер или сервер, перенесите проект вместе с файлом `sklad_requests.db`. Повторно создавать пользователей, назначать роли и подразделения не нужно.

Перед переносом или обновлением сделайте копию базы:

```bash
cp sklad_requests.db "sklad_requests_backup_$(date +%Y%m%d_%H%M%S).db"
```

Если переносите только данные в уже развёрнутый проект, остановите сервер, замените файл `sklad_requests.db`, затем запустите приложение снова.

## Тестовые пользователи

Вход выполняется через `/login`.

| Логин | Пароль | Роль | Доступ |
|---|---|---|---|
| `ivanov` | `ivanov123` | товаровед | только заявки своего подразделения, Ломбард №005 |
| `petrov` | `petrov123` | товаровед | только заявки своего подразделения, Ломбард №126 |
| `driver` | `driver123` | водитель | только маршрутный лист и визиты водителя |
| `admin` | `admin123` | администратор | все разделы, справочники, пользователи, выгрузки |

Администратор создаёт новых пользователей в разделе `Справочники → Пользователи`: ФИО, логин, пароль, роль и подразделение для товароведа. Пароли хранятся только в виде bcrypt-хеша. Для существующего пользователя администратор может задать новый пароль через кнопку `Сбросить`.

Также администратор может:

- редактировать ФИО, логин, роль и подразделение пользователя;
- переводить товароведа в другое подразделение;
- отключать и включать пользователей;
- редактировать название и адрес подразделения;
- отключать и включать подразделения;
- редактировать название, единицу измерения и ориентировочную цену позиции;
- отключать и включать позиции.

Система не даёт отключить или лишить роли последнего активного администратора.

## Защита от плохой связи и повторной отправки

Формы заявок и сохранения доставки защищены одноразовым токеном. Если пользователь нажал кнопку дважды, обновил страницу после отправки или связь оборвалась после того, как сервер уже принял запрос, повторная отправка той же формы не создаст дубль.

При отправке кнопка блокируется и показывает состояние сохранения. После успешного действия приложение возвращает пользователя на рабочий экран и показывает зелёное уведомление.

## Тесты

```bash
pytest -q
```

Тесты проверяют создание активной потребности, частичную доставку, перенос недовоза, объединение новой заявки со старым остатком, запрет доставки сверх остатка, запрет неактивных пользователей/подразделений/позиций, отсутствие частичных изменений при ошибке доставки, запрет повторного сохранения одной позиции в рамках визита и запрет редактирования закрытого визита.

## Администратор: аналитика и выгрузки

В разделе администратора доступны:

- сводные показатели по активной потребности;
- оценка активной потребности в рублях;
- графики топ-подразделений по активной потребности;
- графики топ-позиций по оценке затрат;
- редактирование ориентировочной цены позиции;
- выгрузки CSV:
  - `Сводка CSV`
  - `Потребность CSV`
  - `Заявки CSV`
  - `Доставки CSV`

Оценка затрат считается как `остаток к доставке × ориентировочная цена позиции`. Это управленческая оценка, не бухгалтерская себестоимость.

## Проверка основного сценария

1. Откройте главную страницу и перейдите в роль товароведа.
2. Выберите `Иванов Иван`.
3. В поле поиска подразделения можно ввести номер или часть адреса, например `005`, `Науки`, `Мурино`.
4. Создайте заявку:
   - `Пакеты` — 20
   - `Бумага А4` — 2
5. Вернитесь на экран товароведа: в активной потребности появятся обе позиции.
6. Перейдите в роль водителя, выберите водителя.
7. По умолчанию водитель видит только подразделения с активной потребностью. Через поиск можно найти подразделение по номеру или адресу, а чекбокс `Показать все` включает филиалы без остатка.
8. Откройте нужное подразделение и нажмите `Открыть визит`.
9. Укажите доставку:
   - `Пакеты` — 10
   - `Бумага А4` — 2
10. Нажмите `Сохранить результат`.
11. Проверьте остаток:
   - `Пакеты`: запрошено 20, доставлено 10, осталось 10, статус `частично доставлено`
   - `Бумага А4`: остаток 0, статус `доставлено`
12. Создайте новую заявку товароведом:
    - `Пакеты` — 5
    - `Ручки` — 3
13. Проверьте активную потребность:
    - `Пакеты`: запрошено 25, доставлено 10, осталось 15
    - `Ручки`: запрошено 3, доставлено 0, осталось 3
14. Закройте текущий визит водителя и откройте новый визит в это же подразделение.
15. Водитель должен видеть только актуальный сводный остаток:
    - `Пакеты` — 15
    - `Ручки` — 3

Дополнительно проверьте:

- если ввести доставку больше остатка, система покажет ошибку и не изменит остаток;
- после закрытия визита результат доставки нельзя сохранить повторно.
- количество в заявке и доставке может быть дробным, до 3 знаков после запятой.
